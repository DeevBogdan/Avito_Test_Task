# Тест-кейсы

## Подход

Для покрытия API использованы техники:
- Разбиение на классы эквивалентности.
- Анализ граничных значений.
- Проверки обязательности полей.
- Проверки контрактов ответа.
- Проверки состояния после последовательности запросов.


| ID | Сценарий | Тип | Предусловия | Шаги | Ожидаемый результат | Автоматизация |
| --- | --- | --- | --- | --- | --- | --- |
| TC-01 | Создание объявления с валидным телом | Позитивный | Сервис доступен | Отправить `POST /api/1/item` с валидными `sellerId`, `name`, `price`, `statistics` | `200 OK`, объявление создано, возвращён идентификатор объявления | Да |
| TC-02 | Получение объявления по `id` после создания | Позитивный / E2E | Создано новое объявление | Выполнить `GET /api/1/item/:id` для созданного объявления | `200 OK`, в ответе есть созданное объявление с корректными полями | Да |
| TC-03 | Получение всех объявлений продавца по `sellerId` | Позитивный / E2E | Создано новое объявление с уникальным `sellerId` | Выполнить `GET /api/1/:sellerID/item` | `200 OK`, в списке есть созданное объявление, у всех записей корректный `sellerId` | Да |
| TC-04 | Получение статистики по `item id` | Позитивный / E2E | Создано новое объявление | Выполнить `GET /api/1/statistic/:id` | `200 OK`, статистика соответствует данным, переданным при создании | Да |
| TC-05 | Повторный `GET` объявления по `id` | Corner case | Создано новое объявление | Дважды вызвать `GET /api/1/item/:id` | Оба ответа идентичны, данные не меняются | Да |
| TC-06 | Удаление объявления и повторное чтение | Corner case / E2E | Создано новое объявление | Выполнить `DELETE /api/2/item/:id`, затем `GET /api/1/item/:id` | Удаление успешно, чтение после удаления возвращает ошибку отсутствия объекта | Да |
| TC-07 | Получение объявления по невалидному UUID | Негативный | Нет | Выполнить `GET /api/1/item/not-a-uuid` | `400 Bad Request`, понятное сообщение о невалидном идентификаторе | Да |
| TC-08 | Получение статистики по невалидному UUID | Негативный | Нет | Выполнить `GET /api/1/statistic/not-a-uuid` | `400 Bad Request`, понятное сообщение о невалидном идентификаторе | Нет |
| TC-09 | Получение объявления по корректному, но несуществующему UUID | Негативный | Нет | Выполнить `GET /api/1/item/:uuid`, которого нет в системе | `404 Not Found` | Да |
| TC-10 | Получение статистики по корректному, но несуществующему UUID | Негативный | Нет | Выполнить `GET /api/1/statistic/:uuid`, которого нет в системе | `404 Not Found` | Нет |
| TC-11 | Создание объявления без `name` | Негативный | Нет | Отправить `POST /api/1/item` без поля `name` | `400 Bad Request`, сообщение о пропущенном обязательном поле | Да |
| TC-12 | Создание объявления без `sellerId` | Негативный | Нет | Отправить `POST /api/1/item` без поля `sellerId` / `sellerID` | `400 Bad Request` | Нет |
| TC-13 | Создание объявления без `statistics` | Негативный | Нет | Отправить `POST /api/1/item` без `statistics` | `400 Bad Request` | Нет |
| TC-14 | Создание объявления с отрицательной ценой | Негативный / BVA | Нет | Отправить `POST /api/1/item` с `price = -1` | Запрос должен быть отклонён, `400 Bad Request` | Да |
| TC-15 | Создание объявления с отрицательным `likes` | Негативный / BVA | Нет | Отправить `POST /api/1/item` с `statistics.likes = -1` | Запрос должен быть отклонён, `400 Bad Request` | Да |
| TC-16 | Создание объявления с нулевыми значениями статистики | Corner case / BVA | Нет | Отправить `POST /api/1/item` со значениями `0` в `statistics` | Сервис должен корректно обработать граничные допустимые значения согласно контракту | Нет |
| TC-17 | Проверка контракта ответа `POST /api/1/item` | Контрактный | Нет | Отправить валидный `POST /api/1/item` | Ответ содержит объект созданного объявления с полями `id`, `sellerId`, `name`, `price`, `statistics`, `createdAt` | Да |
| TC-18 | Проверка `Content-Type` JSON в ответах API | Нефункциональный / Контрактный | Нет | Выполнить запросы к основным ручкам | Ответы содержат `Content-Type: application/json` там, где возвращается тело | Частично |
| TC-19 | Проверка времени ответа основных ручек | Нефункциональный | Нет | Выполнить запросы к основным ручкам | Время ответа находится в приемлемых пределах для тестового стенда | Нет |
| TC-20 | Проверка повторного удаления одного объявления | Corner case | Создано и удалено объявление | Повторно вызвать `DELETE /api/2/item/:id` | Поведение должно быть консистентным и соответствовать контракту | Нет |

## Что не автоматизировалось в этой версии

Неавтоматизированные кейсы можно добавить следующим шагом:
- Расширенные проверки контрактов ошибок на всех ручках.
- Дополнительные граничные значения числовых полей.
- Нагрузочные и расширенные нефункциональные проверки.
